Flux Onboarding Handbook

For new engineers joining Flux development. This is the index. Every chapter stands on its own; the recommended path is below.

Philosophy. First six months: read, navigate, understand, ask. Productive code commits come after mental scaffolding is in place. These chapters teach the scaffolding.

Language. All chapters in English. Quoting code, paths, and FTex notes verbatim.

What this handbook is not

  • Not a replacement for Notes\Dev\… and Notes\Tech\…. Those are the source of truth for low-level facts. This handbook orients you so you can read those notes effectively.

  • Not a build guide. Build is in Notes\Dev\1002.BuildingFlux.ftx (with the caveats in chapter 00 §8). On day 1, ask your buddy.

  • Not a feature spec. Specs live in design tickets and PR descriptions.

  • Not API reference docs. Doxygen-style reference does not exist for Flux. Read code; chapter 03 gives orientation.

Conventions used in these chapters

  • File paths are absolute against the repo root (X:\…) or relative under Src\ / Notes\ / Doc\ / Test\.

  • Code excerpts are quoted verbatim from the current tree. If you spot drift, file an issue.

  • Cross-references link by chapter (e.g. "see chapter 02 §7.5"). When section numbers move, update them.

  • Stability badges in chapter 03: 🟢 stable, 🟡 internal, 🔴 in-flight.

  • Single-line callouts (> ⚠️ ...) flag pitfalls and intentional rough edges.

How to keep this handbook honest

These docs degrade fastest at three specific seams. Watch them:

  1. The §8 of chapter 00 ("1002 says vs. today"). Refresh whenever the build toolchain changes.

  2. Stability ratings in chapter 03. Move ratings between green/yellow/red as code matures or refactors.

  3. Real-bug exercises in chapter 09 (especially exercise 5). Replace with newer fixes once a year.

A small ritual: each new joiner files one issue or PR against this handbook reflecting one inaccuracy or one lesson learned. That’s how documentation stays alive.

Companion resources inside the repo

  • Notes\Dev\ — developer notes (FTex). Most important: 1001, 1002, 1003, 1005, 1009, 1011, 1013.

  • Notes\Tech\ — technical notes (FTex). Most useful: 2007 (Curl), 2013 (Flavors), 2015 (Message IDs).

  • Doc\Img\ — domain-specific image galleries. Open Doc\Img\Bend\, Doc\Img\BCell\, Doc\Img\Fold\, Doc\Img\Laser\ to recognize machine vocabulary visually.

  • Help\ — end-user manual (FTex). Sometimes the clearest definition of a feature lives here.

Where to ask

  • Onboarding buddy. Your first day, ask for one. Use them.

  • Issues / PR comments. Public, searchable, durable.

  • Chat (Slack / Teams). Fast but volatile — for high-value answers, repost to the issue tracker.

Contributing to this handbook

The chapters are markdown — edit them like any other source file.

  • Keep prose tight; one idea per paragraph.

  • Don’t repeat content that already lives in Notes\ or Help\ — link to it.

  • Verify every code path you cite — paths and line numbers drift fast.

  • File-scoped ownership: chapter 00 has the broadest reach, so changes there get extra review. Per-subsystem chapters belong to the subsystem owner.

  • All chapters are in English, regardless of contributor language. The codebase is global.
