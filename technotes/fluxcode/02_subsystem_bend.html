Subsystem *Bend* (Press Brake)

Subsystem *Bend* (Press Brake)

Prerequisite: Chapter 00 (repo map) and Chapter 01 (architecture, M/V/Glob, Sheaf, Hub). Goal: Bend is the canonical subsystem. After this chapter you can localize any bug from the press-brake area to the right code and test entry point — without writing a fix. Reading time: 60—​90 minutes. Template note: this chapter doubles as the template for future per-subsystem chapters (Fold, Laser, Punch, Nest, BCell). The same 14-section structure should be followed.

1. TL;DR — Bend in twelve lines

  • Domain: press-brake CAM (classical bending machine: punch + die). The separate domain Fold is the panel bender (TecZoneFold) — don’t confuse them.

  • Code root: Src\Core\Bend\ (47 files, ~1.7 MB of source).

  • Top-level model: Part → BendTech → many BendOp (one bending operation each) and one or more BendSetup (one tool layout on the machine).

  • Auto-tooler: Ops\BendTooler.cs (139 KB) — the central engine that turns a 3D part into a sequence of bends + tool setups + back-gauge positions.

  • Machine: BendMachine (subclass of Machine) — loaded from X:\Data\Machine\<id>\.

  • Tools: BTool (punch / die / holder), per-tech Inventory : BToolList, per-catalog BToolLibrary.

  • Collision: two modules side by side — BendOpCrash.cs (classic) and NBendCrash.cs (new). Both exist simultaneously. Care needed.

  • UI hooks: Src\UI\Bend\BendOpUI.cs (BendOpGlob, BendOpsGlob) and siblings.

  • Tests: Test\Bend\ — one file per year (Bend2017.cs through Bend2025.cs) plus thematic files (BendACB.cs, BendAid.cs, BGPath.cs, BendShuffle.cs, …​).

  • Golden-file pattern: auto-tooler runs, tech.Dump (sw) writes a setup text, Assert.AreTextFilesEqual compares against a .txt reference next to the .geo input.

  • Robotic bend: the logic is not in BCell\ — BendOp itself knows about Robotic cases (regrip, cell stages). BCell models the robot cell, but Bend orchestrates.

  • Important non-obvious: models and VModels (rendering) live together in Core. UI controllers (Glob) live separately in Flux.UI.

3. The data model — class hierarchy

Part
 └─ BendTech                                      [Src\Core\Bend\BendTech.cs]
     ├─ BMachine : BendMachine                    [BendMachine.cs]
     ├─ Inventory : BToolList                     [BendToolLib.cs]
     ├─ Settings : Sheaf (Bend-specific settings)
     ├─ Ops : List<BendOp>                        [BendOp.cs] — the sequence
     │   ├─ Flexes : E3Flex[]                     (1..n, m:n mapping to flexes)
     │   ├─ Phases : int[]                        (one per flex)
     │   ├─ Kind : EBendKind                      (AirBend / Hem / Coining / BumpBend / ...)
     │   ├─ UsedDie, UsedPunch : BTool
     │   ├─ Setup : BendSetup
     │   ├─ BAPos : BendAidPos[]                  (bending-aid positions)
     │   ├─ BGTransition : Sheaf.EBGTransition    (how does the BG move?)
     │   ├─ BumpSteps : int                       (only for bump-bends)
     │   └─ (~200 more properties — file is 210 KB)
     └─ Setups : List<BendSetup>                  [BendSetup.cs]
         └─ Mounts : List<BendMount>              [BendMount.cs]
             ├─ Tool : BTool
             ├─ Pieces : Piece[]                  (segmentation)
             ├─ MountPt : Point3                  (X/Z position on the table)
             └─ IsFlipped : bool

Three subtleties that trip up newcomers:

  1. m:n between flex and BendOp. A hem bend is two phases that exist as two BendOp objects on the same flex. A group bend bundles parallel flexes into one BendOp. See BendOp.cs lines 22—​32.

  2. BendTech is the root, not Part. Part is a generic component that may be laser/punch/fold too. The bend-specific data lives in Part.BendTech.

  3. A BendSetup references Mounts, not Tools. Tool identity is not the unit of granularity for mounts — the same BTool may be mounted at multiple X positions.

5. Lifecycle of a part — sequence diagram in words

What happens internally when a user opens a 3D part and clicks "Bend Auto-Tool"?

1. Part loads (CAD import or .fx file).
   └─ Part.Model contains E3Plane + E3Flex entities.

2. BendTech.Make (part, machine) is called.
   └─ BendTech constructor (BendTech.cs lines 19–120):
       a. RefineColliders() on the model
       b. Split hem phases (one hem → 2 phases)
       c. Identify groupable flexes (CanGroupWith / CanGroupDoubleSheet)
       d. Per group + phase: create one BendOp
       e. ValidOrder() on the list — topological sequence sort
       f. Assign SeqNo
       g. Decouple "flange-in-hole" multi-span bends into a manual list

3. BendTooler.Process (tech, flags) starts the auto-tooler.
   └─ Pipeline (BendTooler.cs lines 48–80):
       a. PrepareForTooling – drop setups, reset sequence
       b. SetupBasicSequence – dead-reckoning order
       c. TurnOnBumpBending – large radii without tools → bump-bends
       d. DemoteCrashingTools – 2D cross-section collisions
       e. UpdateToolScores – favor tools every bend likes
       f. DoToolerCore – final sequence + setup + BG placement

4. Per BendOp: BAPlacer + BGPlacer
   └─ Bending-aid position
   └─ Back-gauge position per op + BG motion path between ops

5. Validation — BendOpStatus + BendOpCrash (or NBendCrash)
   └─ Per-op issues collected as EBATIssue flags.

6. NC code generation runs through `BendTech.GenerateNCCode` → `Hub.GenerateNC`.
   └─ Dispatched by `Src\API\Basis\Basis.cs::GenerateNC` (line 138) to a C# writer
     in `Src\API\Bend\Post\` (TrumpfDLD, Bystronic, LVD, ...) based on the post
     name token in `BMachine.Post`. The XML doc literally says
     *"Generate NC code directly for a part (without using Hex)"*.

7. Common stumbling blocks (observations from the code)

7.1 Two crash engines side by side

BendOpCrash.cs (old) and NBendCrash.cs (new) live together. Both are large (68 + 78 KB). When working on collision logic:

  • Find out which one is active first (settings/flag).

  • Never modify only one without knowing what the other does.

  • This duplication is presumably an in-flight migration.

7.2 The BendTech constructor is the logic

BendTech.Make is a thin wrapper — the real sequence creation lives in the constructor (private, ~250 lines). Unusual but consistent: in Flux, constructors often contain "real" logic because they guarantee a fully constructed object.

7.3 BendOp is a partial class

BendOp.cs (210 KB) holds one part. Grep "partial.BendOp" finds others (e.g. test-related auto-generated parts). *When editing: search the whole solution, not just one file.

7.4 Robotic bend lives in BendOp, not separately

For a bug from the BCell / robotics context (e.g. "regrip at the wrong point"), the logic is in BendOp` (properties `Robotic, RegripDuringBend, IsFirstOp/IsLastOp with cell-stage logic) and in the UI glob (BendOpGlob.AddRegrip in Src\UI\Bend\BendOpUI.cs lines 21—​55). Don’t only search under BCell\.

7.5 Trumpf-specific paths

BendCalc has two implementations: StdBendCalc (default) and TBSBendCalc (Trumpf). The selection happens in BendMachine.BendCalc:

public IBendCalc BendCalc => Settings.BendCalc switch {
   Sheaf.EBendCalc.Default => IsTrumpf ? TBSBendCalc.It : StdBendCalc.It,
   _ => StdBendCalc.It,
};

If a bug only reproduces on Trumpf machines: look at TBSBendCalc.cs and TBSCustomDB.cs.

7.6 Dump format = test contract

If you change BendSetup.Dump or BendTech.Dump, you break every Bend test at once. That’s by design — the dump format is the contract between auto-tooler and tests. Changes here must come with a full golden-file regeneration.

The standard test pattern

[TestFixture (4, "Basic bend auto-tooler tests", Module = "Bend", Subsystem = "AutoTool")]
class BasicBendAuto {
   [Test (82, "Tooling basic part: Bend_04.geo")]
   void Test1 () => Test ("Bend/Auto/Bend_04.geo");

   public static Part Test (string file, int idMachine = 13, /* ... */) {
      Part part = FX.LoadTestPart (file);
      if (part.Model == null) part.DoFoldTo3D ().Is (EResult.OK);
      BendTech.Make (part, BendMachine.Get (idMachine)).Is (EResult.OK);
      var tech = part.BendTech;
      // ...
      BendTooler.Process (part.BendTech, flags);
      CheckDump (tech, outfile);   // ← golden-file compare
      return part;
   }

   public static void CheckDump (BendTech tech, string refFile) {
      string file = FX.TmpTestFile;
      using (StreamWriter sw = new (file)) tech.Dump (sw);
      Assert.AreTextFilesEqual (refFile, file);
   }
}

Three takeaways:

  • Test IDs are globally unique (82, 84, 85, …​) and stable. An ID conflict is a merge problem.

  • Golden-file path mirrors the input — Bend_04.geo ↔ Bend_04.txt.

  • Machine 13 is the default test machine. X:\Data\Machine\13\ holds its definition.

9. The two productively important machine paths

When you want to understand machine behavior:

X:\Data\Machine\13\         ← default test BendMachine
X:\Data\Machine\99\         ← BendMachine.Default in BA.Testing mode
X:\Data\Machine\<custom>\   ← customer machines (delivered with installation)

Per machine:

  • Machine.curl — machine definition (mandatory)

  • Machine.mech — 3D mechanism (animatable)

  • Model\*.iges — 3D parts of the machine

  • BMachine.Post (a string in Machine.curl such as wad:Post/Bend/TrumpfB23.hx) — only a routing token. The basename selects a C# writer in Src\API\Bend\Post\ via Hub.GenerateNC. The legacy .hx extension is preserved for backward compatibility; the file content is not interpreted as Hex.

Note
 BendMachine inherits from Machine (in Src\Core\Model\ or similar). A bend machine is therefore a specialization. The pattern repeats for LaserMachine, PunchMachine, etc.

13. What this chapter does not cover

  • Fold domain (panel bender / TecZoneFold) — separate chapter, similar model but different machine kinematics (clamping beam vs. press beam).

  • BCell (robot cell) — separate chapter, own vocabulary (gripper, suction cups, regrip stations).

  • NC writers per dialect (Src\API\Bend\Post\) — each is a substantial subsystem of its own and a candidate for a dedicated chapter. Note 2004 (Hex) is historical — the postprocessors have all been rewritten in C#.

  • E3Plane / E3Flex geometry model — see Note 1005, separate chapter for Flux.Core\Model.

  • Tool-Setup-Optimizer (TSO) in detail — separate special topic (TSOBatch.cs, Ops\BOptimizer.cs).

14. If you take just one thing from this chapter

Bend is not one operation but a sequence. The sequence is produced jointly by the BendTech constructor and the BendTooler. If a bug is about ordering, tool choice, or setup partitioning, the answer is almost always in Src\Core\Bend\Ops\ — not in the root. The root holds state, Ops\ holds algorithms.
