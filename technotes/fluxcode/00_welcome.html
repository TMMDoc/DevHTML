Welcome & Repo Map

Welcome & Repo Map

Goal of this chapter. After 60 minutes you understand what Flux is, where the code lives, how the modules fit together, and where to read next — without writing a single line of code.

Audience. New engineers joining Flux development. No prior CAM knowledge assumed.

Flux in ten lines

  • Flux is a sheet-metal CAD/CAM application from the Metamation/Trumpf family.

  • Business domains: laser cutting, punching, press-brake bending, panel-bender folding, nesting, bend-cell automation (robot).

  • Stack: C# on .NET 10 (Windows, x64-only), UI in WPF (with WinForms residue), 3D import via HOOPS through a C++/CLI wrapper (Flux.Hula64).

  • One solution: X:\Src\Flux.sln with eight projects (Base, Core, UI, App, API, Console, Migrate, Test).

  • Architecture: Model → View → Glob (Glob ≈ controller). Models and views sit together in Flux.Core; controllers live in Flux.UI.

  • Data formats: geometry/settings as Curl (text) or Au/Gold (binary); read-only assets bundled into Flux.wad. Press-brake NC post-processors are C# classes in Src\API\Bend\Post\ (one per dialect — Trumpf DLD, Bystronic, LVD, Cybelec, Safan, …​); the .hx extension still appears in machine config but is only a routing token now.

  • One executable (Flux.exe), but eight flavors (Dev, Flux, Trumpf, Boost, Engine, B23, Fold, Tech) — selected at runtime via a Flux.flavor text file next to the EXE.

  • Tests: a custom framework called Flit, with [TestFixture] / [Test] attributes. ~169 test files, inputs under X:\TData.

  • Docs: built-in tech notes live in X:\Notes in the proprietary FTex format. Excellent in spirit — and partially outdated (see §8).

  • You don’t have to commit anything productive in the first six months. Read, navigate, understand, ask. That’s what this chapter is for.

1. What is Flux?

Flux is the next-generation CAM stack from Metamation, replacing the older MetaCAM system step by step. It ships under multiple brand names (see §7 Flavors). The header in every source file states:

// Flux ~ Sheet metal CAD/CAM system
// Copyright (c) Metamation India.

The value chain Flux models in code, for a typical part:

CAD import (STEP / IGES / DXF / DWG / ...)
        │
        ▼
Geometry model (B-Rep, mesh, polylines)        ←  Flux.Base + Flux.Core/Model
        │
        ▼
Feature & operation detection                  ←  Flux.Core/Bend, /Laser, /Punch, /Fold
        │
        ▼
Toolpath / NC generation                       ←  Flux.Core/Bend, /Laser, /Punch
        │
        ▼
Simulation & collision                         ←  Flux.Core/Sim, /BCell
        │
        ▼
Post-processor → NC code (G/M-code, BCC, DLD, ...)  ←  C# writers in Src\API\Bend\Post\ (dispatched via Hub.GenerateNC by post name)

Important consequence: Flux is not a CRUD business application. "Model" and "view" are often the same object — for example, an E3Plane knows how to render itself in OpenGL. This point is laid out at length in Notes\Dev\1002.BuildingFlux.ftx lines 31—​61: the original MVC separation was rolled back because it forced ugly workarounds (the so-called Hub whiteboard).

So if it surprises you that a class like LaserCutVM lives in the Core layer — that’s intentional.

5. Architecture pattern: Model → View → Glob

Flux follows an opinionated but consistent MVC variant. The terms appear everywhere — internalize them early:

Layer What Where in code Example classes

Model

Pure domain data

Flux.Core\<domain>\

E3Plane, E2Spline, LaserCut, BendOp

VModel (view)

Knows how to render the model (OpenGL via Lux)

Flux.Core\<domain>\ (same folder!)

E3PlaneVM, E2SplineVM, LaserCutVM, BendMountVM

Glob (controller)

UI shell exposing properties + commands

Flux.UI\<domain>\

BendOpGlob, BendOpsGlob

GElement

Abstract UI description of glob properties

Flux.UI\

 — 

PanelMaker

Translates GElement → real WPF controls

Flux.UI\

 — 

Multi-selection. When the user selects multiple objects, the glob system builds a composite glob showing the shared properties and applying commands to all selections. This is the only clean way to implement bulk edits — don’t iterate selections by hand.

Why this matters. When you add a new domain feature, you almost always touch three places: the model, its VM, and a glob. Miss one, and the UI hook is broken.

Rendering layer. The OpenGL drawing primitives live in Src\Base\Lux\. VModel.Draw() is the common entry; rendering goes through Lux.DrawMesh, Lux.Layer, LuxShading, etc. Each VModel registers a chain of scene-graph nodes (SGNode) and gets a process-wide unique Id once it is "live" on a screen.

8. Build & run — what you actually need to know (current)

⚠️ Important caveat about the tech notes. Notes\Dev\1002.BuildingFlux.ftx is the official build doc, but several sections are outdated. For build / toolchain, trust the current .csproj files and the onboarding-buddy conversation, not 1002 verbatim.

What 1002 says (2017) vs. what holds today:

1002 says today

.NET 4.6.1, C# 6

.NET 10, file-scoped namespaces, #nullable enable (see Src\Core\Flux.Core.csproj)

Mercurial / Kiln (metamation.kilnhg.com)

Git (branch main, GitHub-style PR numbers in git log)

Visual Studio 2015

recent Visual Studio (Solution format version 16, platform x64)

Any-CPU build with 32/64-bit Hula trick

x64-only (see <PlatformTarget>x64</PlatformTarget> in every csproj)

FluxPreSetup.2017.A.exe

newer pre-setup variant (ask internally)

What still holds:

  • Three subst drives are standard: X: (code/data), W: and N: (Trek). Paths in scripts often hard-code X:\….

  • Environment variables: FLUXDATA=X:\Data, FLUXTESTDATA=X:\TDATA, FLUXDEVELOPER=1, FLUXWAD=X:\Wad, TREKPATH=N:\Bin (for post-processors / code-finishers from the Trek stack).

  • A separate solution (Hula64.sln in Src\Hula\) for the HOOPS wrapper — don’t build it, copy the binary from Resources\Hoops\.

Concrete first build attempt:

  1. X: set up as a subst (subst X: C:\Work\Flux or similar)?

  2. Environment variables set?

  3. HOOPS DLLs copied to Src\bin\hoops\?

  4. Open Src\Flux.sln, build *Release

    x64*.

  5. Run Flux.exe from Src\bin\ (should detect Dev flavor because no Flux.flavor file is present).

If something breaks: ask buddy first, then read 1002, then search internal issues — in that order.

9. Code conventions (what you’ll see while reading)

From Notes\Dev\1001.FluxDevStyleGuide.ftx and observation of current code:

  • 3 spaces for indentation (no tabs). Yes, three. "Goldilocks number."

  • K&R style with brace omission for single statements:

    if (nLevel > 0 && !mLevelStack.IsEmpty)
   PopLevel ();

  • Spaces before ( and {, no space before [, no space inside ().

  • File header in every .cs file (the stylized ASCII hammer at the right margin is the internal logo — please keep it when adding new files).

  • Modern C# 10+ in new files: namespace Flux; (file-scoped), #nullable enable.

  • Class / file headers span the full width (column 100); region headers per top-level class.

Real example (Test\Bend\BasicBendAuto.cs):

namespace Flit;
#nullable enable

[TestFixture (4, "Basic bend auto-tooler tests", Module = "Bend", Subsystem = "AutoTool")]
class BasicBendAuto {
   [Test (82, "Tooling basic part: Bend_04.geo")]
   void Test1 () => Test ("Bend/Auto/Bend_04.geo");
   ...
}

10. Tests — what they look like

  • Framework: Flit (custom-built, not NUnit/xUnit). Attributes: [TestFixture(id, description, Module=..., Subsystem=...)] and [Test(id, description)].

  • Inputs: .geo files under X:\TData\<domain>\….

  • Organization: one directory per domain under Test\ (e.g. Test\Bend\, Test\Laser\). Inside that, often per-year regression bundles: Bend2017.cs, Bend2018.cs, …​, Bend2025.cs. Each year-file is a layer of historically collected bugs.

  • Reading tip: to find a recent feature in Bend, look at Test\Bend\Bend<currentYear>.cs first — that’s where regression tests for PR-described issues land.

A 15-minute reading tour is described in detail in chapter 06.

12. First reading exercise (~30 min)

Task: in at most half a page, explain what happens when the user edits a Bend setup through the UI.

Approach (read-only, no code change):

  1. Grep for class BendSetup — where does the class live?

  2. Find the matching glob in the UI layer.

  3. Trace one BendSetup property edit: what methods get called?

  4. Where is the change persisted? (Hint: Curl, Au, FX.)

  5. Is undo registered? Where?

Rubric (for your mentor):

Level Criterion

1

Finds BendSetup, knows roughly which domain it lives in

2

Also finds the UI controller, can sketch UI → model data flow

3

Plus: locates persistence and undo, can explain VModel’s role

4

Spots an edge case (e.g. multi-selection: how does the composite glob behave?) and backs the answer up with code/tests

More exercises in chapter 09.

14. If you take just one thing from this chapter

Flux is intentionally not pure MVC. Models, views (VModel), and controllers (Glob) live in different assemblies, but model + view sit together in Flux.Core. If that surprises you, read Notes\Dev\1002.BuildingFlux.ftx lines 31—​61 — it was a deliberate decision against the previous architecture.

The rest you can look up.
