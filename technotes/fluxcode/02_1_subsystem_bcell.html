Subsystem *BCell* (Bend Cell Automation)

Subsystem *BCell* (Bend Cell Automation)

Prerequisite: Chapter 02 (Bend) read. BCell is layered on top of Bend — every BCell concept ultimately wraps a BendOp. Goal: understand the bend-cell automation subsystem well enough to localize any robot-cell bug to the right code area without writing a fix. Reading time: 60—​90 minutes. Companion to: chapter 02 §7.4 ("Robotic bend lives in BendOp, not separately") — that paragraph is the orientation; this chapter is the depth.

1. TL;DR — BCell in twelve lines

  • Domain: automated press-brake bending performed by a robot cell (Trumpf BendMaster 60/150, Cell5000, Cell7000, FlexCell 10, B23-attached cells).

  • Code root: Src\Core\BCell\ (34 files, ~1.2 MB) plus Src\UI\BCell\ (10 files).

  • Top-level model: BCellTech is abstract — concrete subclasses include BM5000/BM7000/BC7000 family. Lives at BendTech.Cell (every BendTech may have one cell).

  • Three primary stages: Pickup (BCPickup) → many Bends (regular BendOp plus possibly BCRegrip between them) → Deposit (BCDeposit).

  • Cell items: physical objects that can sit in the cell — BPallet, BDispenser, BCupGrip, BJawGrip, BProfileGrip, BFlipConveyor, BTiltConveyor, BLightBox, BFurniture — all under one base BCellItem.

  • Components vs. items: an item is the catalog object; a BCComponent is the placed instance in this cell at a specific transform.

  • Robot pose: kinematic state via BRobotPose (axes / wrist / part transforms); inverse kinematics included.

  • Routing: BRouting decides per-bend insert/bend/extract strategy (front, side-left, side-right, with-ram, vacuum-on, swivel).

  • Regrip flavors: BCRegrip.EFlavor — Pickup, Deposit, Station, Clamp, PostBend, DuringBend, CenteringTable, DoubleSheet. Each has different rules about where it can be added.

  • Output: robot code (proprietary text) + EIN files (Trumpf simulation) + standard NC for the brake.

  • UI controllers: BCellGlob, BPickupGlob, BRegripGlob, BDepositGlob, BWaypointGlob, BGripperGlob, BJawPickupGlob, BGripperToolGlob, BFlexiGripperGlob, BRobotStrategyGlob — all under Src\UI\BCell\.

  • Tests: Test\Bend\Cell\ — yearly bundles (Cell2021.cs …​ Cell2026.cs) plus thematic (CellBasic.cs, CellFixes.cs, CellGripper.cs, BCRegripTest.cs, BCPickupTest.cs, BCDepositTest.cs, BCRouting.cs, BCExtract.cs, CellCustomer.cs, CellUI.cs).

Three subtleties

  1. BCellTech is abstract. Every concrete cell is one of a small set of subclasses (BM5000 family, BC7000Tech). When you read BCellTech.Foo(), look at the concrete class that the part actually owns — overrides matter.

  2. Items are catalog, components are placements. Editing a BCupGrip (item) potentially affects all parts that use it; editing its BCComponent (placement) only affects the current cell.

  3. Stages are siblings, but ordered. A cell run is Pickup → (Bends with optional Regrips between them) → Deposit. The Bend ops themselves are owned by the parent BendTech; BCell owns the inter-bend "glue".

5. The eight regrip flavors (the most subtle topic)

BCRegrip.EFlavor (in BRegrip.cs) drives where and how a regrip can be inserted. Each flavor has different placement rules — see BendOpGlob.AddRegrip in Src\UI\Bend\BendOpUI.cs lines 21—​55.

Flavor Where it occurs Notes

Pickup

Right after the pickup, before the first bend

Available if the pickup pose differs from what the first bend wants.

Deposit

Right before deposit, after the last bend

Available if the deposit pose differs from the last-bend hold.

Station

At a fixed regrip station between two bends

Requires HasRegripStns and not jaw-grip.

Clamp

While the part is held by the brake clamp

Available if previous regrip wasn’t a station/pickup/centering-table.

PostBend

Right after a bend, before the next

Cannot stack with another Station-type regrip.

DuringBend

While the part is being bent (rare, machine-dependent)

Requires RegripDuringBend = true on the prior op.

CenteringTable

At a centering-table component

Requires a Centering-flagged component.

DoubleSheet

At a double-sheet station for paired processing

Requires HasDoubleSheetStn.

When debugging "regrip in the wrong place" or "regrip cannot be added", the answer is almost always in this enum’s rules combined with the per-cell capability flags.

6. Lifecycle of a robotic part — sequence in words

1. Part loads (.fx or 3D import).
2. BendTech.Make creates the bend projection (chapter 02 §5).
3. If the target machine is cell-attached:
   a. BCellTech is constructed (concrete subclass: BM5000 / BC7000 / Fanuc).
   b. Cell components (gripper, pallet, dispenser, regrip stns) are placed.
   c. BendTech.Cell = cellTech.
4. BendTooler.Process runs (still chapter 02).
   ─ But: now mRobotic = true → tool-scoring / sequencing rules differ.
5. Cell-level expert passes:
   a. PickupExpert / DispenserExpert decide pickup parameters.
   b. BRegripExpert decides regrips between bends.
   c. BDepositExpert decides the deposit lattice.
   d. CellRefine refines the plan.
   e. RPathPlanner plans robot motion between every pose.
   f. BStepComputer materializes per-step waypoints.
6. BSimStepNeo simulates the run end-to-end.
7. Validation: per-stage EBCStatus / EBStatus flags collected.
8. Output:
   ─ NC code for the brake — produced by a C# writer in Src\API\Bend\Post\
     (e.g. BNCWriter for Cell5K / TrumpfB23, TrumpfDLD, ...) via Hub.GenerateNC.
   ─ Robot code (cell-specific text) — generated by BCellTech.RobotCode + MangleRobotCode.
   ─ Optionally an EIN file (Trumpf simulation) via BCellTech.EINWriter.

8. Common stumbling blocks

8.1 BCell logic is split across BendOp and BCRegrip

Robotic-bend awareness lives partly in BendOp (Robotic, RegripDuringBend, IsFirstOp/IsLastOp cell-stage logic) and partly in BCRegrip (the regrip itself). When a "regrip in wrong place" bug arrives, both places need inspection, not just BCell.

8.2 BCellTech is abstract — every override matters

BCellTech defines the contract; BM5000, BC7000Tech, BCFanucTech override behavior. A bug "only on Cell7000" usually points at 7K\BC7000Tech.cs. A "only on Fanuc" bug → Fanuc\BCFanucTech.cs.

8.3 Versioning is everywhere

BPickup.PostLoad (BPickup.cs lines 31—​50) does an in-line migration based on mCell.FXBuildRev and mCell.FXBuildNo — not just AuType.ObjVersion. The pattern is: turn off status flags that older builds couldn’t set correctly. When you find a flag mysteriously missing, check the build-number gates.

8.4 Robot-code post-processing is in the dump

BCellTech.MangleRobotCode (lines 80—​119) post-processes the raw robot code to produce the canonical text used in tests. If you see odd DZ/DY/DX deltas in golden files, that’s this method emitting compact relative motion.

8.5 Cell items have EFlags for capability gating

BCellItem.EFlags (25 flags!) gates compatibility (BM60 vs. BM150 vs. B23 vs. FC10 vs. Cell7000 vs. TBC7030, plus structural attributes like Conveyor, Centering, DoubleSheet, Inclined, RotateBy180). When a component doesn’t appear in a placement dialog, the answer is here.

8.6 The IGripSource interface

BCPickup and BCRegrip both implement IGripSource (search the interface in BCupGrip.cs or sibling) — they are the two places the part transitions to a new gripper hold. Bugs about "gripper hold transform after regrip/pickup" are at this interface.

8.7 Two output streams

A cell test uses two golden files: a .txt (the cell dump) and a .nc (the brake NC code). Changing one without the other = one of the two will fail.

From Test\Bend\Cell\CellBasic.cs lines 131—​137:

public static Part Test (string infile, int idMC = 93, bool checkBNC = true) {
   var part = FX.LoadTestPart (infile);
   part.DoSwitchToBending (EDoFlags.NoGAPAdjust, BendMachine.Get (idMC));
   part.BendTech.Cell.CheckDump (Path.ChangeExtension (infile, ".txt"));
   if (checkBNC) part.BendTech.CheckBNC (Path.ChangeExtension (infile, ".nc"));
   return part;
}

Note the differences from chapter-02 Bend tests:

  • idMC = 93 is the standard cell-attached machine (X:\Data\Machine\93\), not 13.

  • DoSwitchToBending not BendTech.Make directly (cell-aware switching).

  • Two golden checks: Cell.CheckDump (cell text) + BendTech.CheckBNC (brake NC).

  • BendCellMisc adds EIN-file tests (EINTest, lines 169—​174) and recognition-image tests (Test, lines 175—​187).

Test inputs live under X:\TData\Cell\<area>\ with the same subdirectory taxonomy as the test categories above. Customer cases use the prefix C<number> (e.g. C45292, C42921); internal tickets use T<number> (e.g. T4715, T8574).

14. What this chapter does not cover

  • NC code generation for the brake — see chapter 02 §10. NC writing is handled by C# writers in Src\API\Bend\Post\ (e.g. BNCWriter for Cell5K / TrumpfB23, TrumpfDLD, …​) dispatched via Hub.GenerateNC.

15. If you take just one thing from this chapter

BCell lives at BendTech.Cell and orchestrates three stages — Pickup, Regrips, Deposit — around the existing Bend sequence. Almost every cell-specific bug is a flavor / capability / version gate in one of those three stages, or the regrip-add rules in BendOpGlob.AddRegrip. Models are catalog (BCellItem); placements are instances (BCComponent). Always check both.
