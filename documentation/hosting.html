Repositories and Workflow

Repositories and Workflow

There are two organizations in GitHub for managing document control, translation and branding.

  • Metamation

  • TMMDoc

    1. The Metamation contains master files for all software products kept in a most complete form without any branding information. - all repositories are private

    2. The TMMDoc contains HTML files for all software products with branding and translation. - all repositories are public

    3. Translation is done from these master files to keep the translation clean and non redundant. Each release cycle we get the diff from the master file and only send the delta for translation.

    4. Within the branding process we only have delta descriptions in contents.txt. This ensures that every added file in the master document gets automatically populated in all brands.

Example - FluxBook repository

Workflow for collaboration with documentation

Product Owners

Product owners are responsible for their own product documentation. Product owners should regularly check their documentation to ensure it is complete and readable. Product owners must evaluate the release notes, open a case in Fogbugz, establish priorities, and notify the documentation team.

  • Minor updates (like updating screen shots) don’t need a lengthy description. Just create a Fogbugz case for Documentation with reference to the relevant release note.

  • Major updates and new features require Product Owners to define the substance of the documentation. It is needed to include high-quality images (.png files) with the input files. Provide the input in asciidoc format (if possible) or in a basic word document for the documentation team to work with.

  • Minute updates can be added directly in the respective product’s GitHub repository.

Create a Fogbugz case to inform the documentation team about this minor update.

Procedure for minor updates in GitHub:

  1. Navigate to the file you want to edit.

  2. Click on Edit this file icon and edit your changes.

  1. After editing, click Commit changes…​ to create a new branch with a commit message.

  1. Type the commit message, add description, select Creating a new branch for this commit, and type the branch name.

  1. Click Propose changes to create a pull request.

  1. Goto the Pull Request and assign it to documentation team member.

  2. After the documentation team has reviewed and merged the changes into the main branch, close the Fogbugz case.

Documentation Team

The documentation is responsible for

  • Updating the documentation according the Fogbugz cases created by the Product Owners.

  • Work on major updates (conver input file to asciidoc format), commit changes and assign pull request for review.

Structure of TMMDoc

The FluxBook repository contains several modules (CAD, CUT, BEND, FOLD etc.,) as folders along with the translated versions (EN, DE, ES etc.,) in individual folders.

A example folder structure of TecZoneLaser is shown below:

Branding Folder

For each branding (let’s say TechZoneLaser) a separate Manifest and contents file manifest.txt and contents.txt files.

Manifest.txt folder

Manifest.txt file looks like this:

TITLE=TecZoneLaser
OUTDIR=../TecZoneLaserHTML
LOGO=img/Trumpflogo.png,207,34
COPYRIGHT=Copyright &copy; 2023 by Trumpf. All rights reserved.
LANGUAGES=EN,DE,ES
ATTRIBUTE=appName:Flux
ATTRIBUTE=bendApp:Flux

For the branding a new title etc is given, languages are added

Contents.txt folder

The contents.txt file is a substraction and replacement description. if it is a empty file then the direct version of the Master is published. If a subproduct like TecZoneLaser is extracted it might look like this:

FluxBook contents.txt

TecZoneLaser contents.txt

[Intro]
[CAD]
[Cut]
[Bend]
[BCell]
[Fold]
[Settings]
[Report]
[Appendix]

[.\Cut\intro]
[HomeZone]
[.\Cut\workflow]
[.\Cut\Advancetopics]
[.\Cut\Applicationsettings]
-[Cut]
-[CAD]
-[Bend]
-[BCell]
-[Fold]
impressum

Explanation of contents.txt in the TecZoneLaser folder:

[intro]

The intro chapter is used but heavily modified. The master file has a lot of bending examples which are not good for a laser program.

[HomeZone]

A new chapter HomeZone is added which is specific for the Trumpf installation of how FluxCut is integrated in Boost HomeZone

[.\Cut\workflow] etc…​

The folders workflow, advancedtopics and Applicationsetup is moved up one level.

-[Cut]

Since all folders were lifted up we don’t need the original cut chapters. Applicable also for CAD, BEND, BCELL, FOLD folders.

impressum

The Trumpf impressum is added.

Further details of the folder structure

  1. The branding folder needs to have the same structure as the master folder. Cut is a sub-chapter of FluxBook which will be lifted one level up so it needs to be on level 2.

  2. Res folder has informations required for Bento to create HTML file to define the branded color scheme.

    1. bento.css

    2. site.css

    3. favicon.ico

  3. As stated the contents.txt of e.g. .\TecZoneLaser\Cut\workflow is a delta file, if its empty then the chapter is taken as it is or if a subfolder is not set up the master subfolder will be taken as it is.

  4. If e.g .\TecZoneLaser\Cut\ApplicationSetting\index.adoc needs to be changed the path needs to be replicated in the branding folder.
